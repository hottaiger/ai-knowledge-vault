# Codex 学习手册 — CodexGuide 核心要点

> 来源：https://codexguide.ai | 作者：canghe | MIT Licensed

---

## 一、学习路线总览

CodexGuide 按 **四层** 组织学习路径：

| 阶段 | 目标 | 验收标准 |
|------|------|----------|
| **入门** | 跑通一个低风险任务 | 能用桌面 App/CLI 完成完整任务闭环 |
| **进阶** | 形成稳定任务方法 | 能写清目标、范围、约束、验证和交付 |
| **工程化** | 进入真实项目流程 | 每次改动都有 diff、测试结果和风险说明 |
| **团队化** | 沉淀规则和案例 | 项目有 AGENTS.md、案例库、排障手册 |

### Codex 的五个入口

- **CLI**：贴近终端和本地仓库，适合快速修改、运行命令、排障
- **桌面 App**：适合多任务、Skills、Automations、子代理、插件化工作流
- **Web/Cloud/ChatGPT**：适合连接仓库、分派长任务、并行处理、生成 PR
- **IDE**：适合编辑器上下文的解释、修改、审查
- **集成生态**：GitHub、Slack、Linear、MCP 等

## 二、桌面 App 核心概念

### 设置项速查

| 设置 | 作用 | 新手建议 |
|------|------|----------|
| 常规 | 工作模式（编程/日常）+ 权限级别 | 编程模式 + 最小权限 |
| 个性化 | Codex 沟通风格和默认偏好 | 只写 3 条以内 |
| 配置 | 模型、推理强度、沙盒、审批、网络 | 默认即可 |
| MCP | 连接外部工具（浏览器、数据库、飞书等） | 一次只开一个 |
| 钩子 | 任务前后自动触发脚本 | 先空着 |
| Git | 仓库、分支、diff、提交协作 | 任务前后看 git status |
| 环境 | 依赖、命令、环境变量 | 先只记录启动命令 |
| 工作树 | Git worktree 并行任务 | 会用分支后再用 |
| 浏览器 | 打开网页、点击、截图 | 先只用于本地预览 |
| 电脑操控 | 读取屏幕、点击应用 | 只做观察和截图 |

### 权限模式选择

- **默认权限**：可读写当前工作区文件
- **自动审核**：对额外权限请求部分自动判断
- **完全访问**：能力最大，风险也最大

> ⚠️ 新手不要长期开完全访问，按任务临时使用。

## 三、CLI 安装与使用

### 安装

```bash
npm install -g @openai/codex
# 或更新
npm install -g @openai/codex@latest

# 验证
codex --version
```

环境要求：macOS 12+ / Ubuntu 20.04+ / Windows 11 WSL2，Git 2.23+，内存 4GB+。

### 登录

```bash
codex  # 按提示完成浏览器登录
```

### 第一个只读任务

```
请先阅读这个仓库的目录结构、README、包管理器配置和测试配置。不要修改文件。请总结：
1. 项目用途
2. 主要技术栈
3. 如何安装依赖和运行测试
4. 你建议我下一步交给你的 3 个低风险任务
```

### 常见安装问题

| 现象 | 处理方式 |
|------|----------|
| codex 命令找不到 | `npm bin -g` 看路径，加入 PATH |
| 登录后仍无权限 | 检查账号计划 / 重新登录 |
| Windows 异常 | 使用 WSL2 |

## 四、任务执行机制

### 顺序执行

- 同一对话内任务排队执行，后来的命令需等待
- **插队操作**：点击引导选项可以让新要求直接插入当前执行中的任务

### 并行执行

- 在同一个 Project 下新建多个对话，各对话任务并行互不干扰
- 适合：一个修 bug、一个改文档、一个跑测试同时进行

## 五、AGENTS.md — 项目规则

> AGENTS.md 是 Codex 的记忆文件，每次新对话 Codex 会自动读取它作为上下文。

### 为什么需要

没有 AGENTS.md 时，Codex 需要反复推断：用什么包管理器、怎么跑测试、哪些文件不能改、提交前跑什么检查。写下来就省去反复解释。

### 作用域

- **项目级**：项目根目录的 `AGENTS.md`，仅对当前项目生效
- **全局级**：桌面 App → 设置 → 个性化 → 自定义指令，对所有项目生效

### 最小可用模板

```markdown
# AGENTS.md

## 项目命令
- 安装依赖：`pnpm install`
- 本地开发：`pnpm dev`
- 构建：`pnpm build`

## 改动规则
- 修改前先阅读相关文件
- 保持现有代码风格
- 不提交构建产物和环境变量文件

## 验证要求
- 文档改动运行：`pnpm build`
- 代码改动运行相关测试

## 安全边界
- 不读取 `.env` 或任何私有凭据
- 不执行发布、部署、数据库迁移和删除数据命令
```

### 写作建议

- 越具体越好：「`pnpm test`」>「记得测试」
- 写清楚生成目录、构建产物、锁文件策略
- Monorepo 要说明每个包的边界
- 安全敏感项目单独写「禁止事项」

## 六、实践方法

### 三条底线

1. **任务要可验证**：没有验证方式的任务难判断质量
2. **范围要可控制**：越靠近核心业务/权限/支付越要小步
3. **结果要可复盘**：每次任务留下「原因、改动、验证、风险」

### 最小闭环模板

```
请处理：[明确目标]

背景：
- [当前现象/业务上下文]

范围：
- 可以修改：[文件/目录]
- 保持不变：[文件/目录/行为]

约束：
- 不引入新依赖
- 保持公开 API 兼容
- 不处理生产数据或敏感凭据

验证：
- 运行：[命令]
- 如果失败，请定位原因并说明阻塞

交付：
- 改了什么
- 为什么这样改
- 验证结果
- 剩余风险
```

## 七、配置体系（四层）

| 层级 | 文件/入口 | 解决的问题 |
|------|-----------|-----------|
| 项目规则 | `AGENTS.md` | Codex 理解仓库约定、命令、边界 |
| 本地配置 | `~/.codex/config.toml` | 模型、沙盒、审批、profile |
| 扩展能力 | Skills、MCP、Plugins、Subagents | 复用流程、连接外部工具 |
| 安全边界 | Sandbox、Approvals、Admin policy | 管控命令、网络、敏感文件 |

### 推荐配置路线

- **第 1 天**：装 CLI → 登录 → 只读总结 → 创建 AGENTS.md → 学会 `/status` `/model` `/approvals` `/diff` `/quit`
- **第 1 周**：写好测试命令和代码风格 → 配置 `config.toml` → 沉淀任务模板 → 记录失败案例
- **第 1 个月**：团队规则进仓库 → 流程沉淀成 Skills → 设置审批要求

### 常见误区

| 误区 | 更好做法 |
|------|----------|
| 只调模型不写上下文 | 明确仓库结构、任务范围、验收标准 |
| 只改代码不让验证 | 每个任务指定最小验证命令 |
| 直接放开全部权限 | 先用只读和审批模式 |
| 把规则塞进提示词 | 长期规则放 AGENTS.md 或 Skill |
| 团队成员各配各的 | 关键规则进仓库，个人偏好留本地 |
