## 2. 高价值结论

### 2.1 设计改了，不是“重跑设计”，而是“刷新基线”

深度设计做完后又修改设计，不一定要把整个 `comet-design` 重来一遍。标准动作是：

1. 修改设计源文件
   - `proposal.md`
   - `design.md`
   - `tasks.md`
   - `docs/superpowers/specs/...design.md`
2. 重新生成 design handoff
3. 必要时重新跑 design guard
4. 再回到 `build`

示例命令：

```bash
COMET_ENV="${COMET_ENV:-$(find . "$HOME"/.*/skills "$HOME/.config" "$HOME/.gemini" -path '*/comet/scripts/comet-env.sh' -type f -print -quit 2>/dev/null)}"
. "$COMET_ENV"
bash "$COMET_HANDOFF" flagship-store-car-price design --write
bash "$COMET_GUARD" flagship-store-car-price design --apply
```

注意：如果当前状态已经是 `build`，而 handoff 只允许在 `design` 阶段生成，就要先把 phase 切回 `design` 再生成。

### 2.2 handoff 的本质：设计阶段给后续阶段的“标准交接包”

`OpenSpec handoff` 不是为了存档好看，而是为了把“当前设计文档里的事实集合”冻结下来，交给后续 `build` / `verify` 阶段使用。

它至少解决三件事：

- 给后续阶段一个统一输入，而不是靠聊天记忆继续做
- 生成一份可追溯的上下文快照
- 用 hash 判断“设计是否改过但没同步交接包”

常见产物：

- `openspec/changes/<change>/.comet/handoff/design-context.md`
- `openspec/changes/<change>/.comet/handoff/design-context.json`

### 2.3 “上下文基线”是什么意思

基线就是：

> 从这一刻开始，后续实现、验证、讨论默认都依赖的那一版事实集合。

这版事实通常来自：

- `proposal.md`
- `design.md`
- `tasks.md`
- `specs/*/spec.md`
- `docs/superpowers/specs/...design.md`

handoff 的作用是把这些内容冻结成“当前有效版本”。后面谁来做，都应该默认按这版来，而不是按记忆或旧聊天记录来。

### 2.4 基线如何约束后续工作

它至少从 5 个层面约束后续阶段：

1. 约束范围  
   后续不能“顺手扩需求”。

2. 约束实现方式  
   已经确认过的设计决策不能被随意替换。

3. 约束验证标准  
   `verify` 不是随便验，要按设计里写过的验收点验。

4. 约束讨论引用版本  
   后续讨论默认引用 handoff 对应的那版文档事实。

5. 约束状态机一致性  
   文档改了但 handoff 没刷新，guard 可以拦住。

### 2.5 不重新生成 handoff 就直接 build，会有什么问题

最直接的风险是：

> 文档已经变了，但后续阶段仍然拿着旧交接包在做事。

可能后果：

- 新设计要求没进入实现
- 实现按旧方案继续走
- verify 按旧标准验新代码
- guard 发现上下文不一致并阻塞流转

一句话：

> 你改了设计说明书，但没有把新版说明书正式交给后续阶段。

### 2.6 build guard 失败，不一定是代码错，可能是“仓库默认构建命令不存在”

这次实战里，`build guard` 一度卡在：

```text
[FAIL] Build passes
```

根因不是业务实现失败，而是 Comet guard 默认会找：

- `.comet.yaml` 里的 `build_command`
- 否则找 `package.json` 的 `"build"` 脚本

而该仓库没有默认 `npm run build` 可用，所以需要显式写：

- `build_command`
- `verify_command`

示例思路：

```yaml
build_command: ./node_modules/.bin/eslint ... && yarn test ...
verify_command: ./node_modules/.bin/eslint ... && yarn test ...
```

这说明：

> Comet 的 phase guard 依赖“项目级可执行校验入口”，如果仓库没有通用 build 脚本，就需要手动声明。

### 2.7 为什么主工作区和 worktree 看到的 `tasks.md` 不一样

因为在 `worktree + subagent` 模式下，真正开发、打勾、状态推进都发生在 worktree 分支里。

所以会出现：

- 主工作区 `tasks.md` 还没勾
- worktree 里的 `tasks.md` 已经全部勾完

这不是流程错，而是因为：

> 你看到的是两份不同工作副本中的同一路径文件。

要判断当前 Comet 真正状态，优先看：

- worktree 的 `openspec/changes/<change>/.comet.yaml`
- worktree 的 `tasks.md`

### 2.8 为什么 build 阶段应该让用户选隔离方式

在 Comet 里，`build` 不是默认就往当前工作区开工。  
正常应先让用户明确选择：

- `branch`
- `worktree`

再选择执行方式：

- `subagent-driven-development`
- `executing-plans`

如果代理提前替用户推进，没有停在决策点上，就是流程执行过快，不够标准。

## 3. handoff 文件里，约束是怎么体现的

以 design handoff 为例：

- `design-context.md`
- `design-context.json`

### `design-context.md`

偏向“给人看的交接正文”。

里面会摘录：

- `proposal.md`
- `design.md`
- `tasks.md`
- 相关 `spec.md`

并标记：

- `Source`
- `Lines`
- `SHA256`

这意味着后续不是抽象地“参考 design”，而是明确到：

> 参考这个文件的这一段内容，这一版 hash。

### `design-context.json`

偏向“给脚本和状态机看的机器索引”。

典型会包含：

- `change`
- `phase`
- `files`
- 每个源文件的 `sha256`
- 整个 handoff 的 `context_hash`

这里最关键的是 hash：

- 文件 hash 变了，说明源文档变了
- `context_hash` 变了，说明交接基线变了

所以它能支撑 guard 做一致性检查。

## 4. 关于验证：UI 接入层先实现后补验证，还有没有价值

有价值，但性质变了。

如果是“先测试后实现”，测试的价值更偏：

- 驱动接口设计
- 暴露依赖边界问题
- 逼出更清晰的实现结构

如果是“先实现后补验证”，测试的价值更偏：

- 验收证据
- 回归保护
- 防止后续改坏关键路径

所以结论不是“没用”，而是：

> 它仍然需要，只是从“测试驱动”退化成了“结果兜底”。

## 6. 一个简化的心智模型

可以把 Comet 理解成 4 层：

1. 文档层  
   `proposal/design/tasks/spec`

2. 交接层  
   `handoff md/json`

3. 状态层  
   `.comet.yaml`

4. 守卫层  
   `comet-guard.sh`

它们之间的关系是：

```text
设计文档
  -> 生成 handoff
  -> 写入状态 hash
  -> guard 用状态和命令检查是否可以进入下一阶段
```

一旦文档变了但 handoff 没变，这 4 层就会失配。

## 7. 实战建议

- 改设计后，优先想“要不要刷新 handoff”，而不是“要不要重跑整个 design”
- 进入 build 前，明确隔离方式和执行方式
- 在 `worktree` 模式下，所有状态判断都优先看 worktree，不看主工作区
- 如果仓库没有通用 `build` 脚本，尽早给 `.comet.yaml` 配 `build_command`
- `tasks.md` 的勾选不要靠印象，要和实际 worktree 文件、验证证据同步
- “先实现后补验证”可以接受，但要清楚那是回归保护，不是强 TDD

## 8. 一句话总结

Comet 的关键不只是“分阶段”，而是：

> 用 handoff 把设计冻结成基线，用 `.comet.yaml` 记录当前阶段，用 guard 检查文档、状态和验证命令是否仍然一致。

一旦理解了这三者的关系，Comet 的大部分“为什么要这样做”就会变得很清晰。

## `docs/superpowers/specs/2026-06-10-flagship-store-new-car-hide-components-design.md` —— Design Doc

**作用**：技术设计的"单一事实源"，build 阶段所有改动都按它执行。

**build 阶段具体怎么用**：
- §1 "Scope" 明确**改 / 不改**的范围（"包含什么 / 不包含什么"）
- §2 "Architecture" 给出**精确行号**（NewEnergy 711-815、FuelVehicle 678-805、变量在 186 行）

- "Spec Patch" 章节列出**要回写 spec 的具体改动**（子 Spec §2.1 增 `FirstScreenPrice`、总索引 §3 矩阵 14 行取值规则）—— build 时直接照抄
- "Risks / Trade-offs" 提前说明风险与缓解措施（如 selector re-render、TS 类型 import），build 时自检
- "Test Strategy" 明确**不写单测**，用 `openspec validate` + 人工回归替代

**实际使用证据**：本次 build 的 Task 1+2 的精确行号、包裹形态、selector 写法，Task 3-5 的 spec 同步清单，Task 6 的验证命令，**全部从 Design Doc 直接复制**。Plan 是把 Design Doc 拆成可执行步骤，Design Doc 是**为什么这么做**。

## `docs/superpowers/plans/2026-06-10-flagship-store-new-car-hide-components.md` —— Implementation Plan

**作用**：Design Doc 到具体代码改动的"中间表示"，把技术设计**拆成可机械执行的 task**。

**build 阶段具体怎么用**：
- **frontmatter 元数据**（`change` / `design-doc` / `base-ref`）—— 标识 Plan 关联到哪个 change、引用哪个 Design Doc、跨多少提交（`base-ref=0196c01a…` 用于 verify 阶段统计改动规模）
- **6 个 Task 拆分**（NewEnergy 短路 / FuelVehicle 短路 / 旗舰 Spec 同步 / 总索引状态升级 / §3 矩阵补齐 / 验证与回归）—— 每个 Task 是一个**独立可提交**的改动单元
- **每 Task 内的 Steps**（写代码 → 跑 tsc → commit）—— **TDD / bite-sized 粒度**（2-5 分钟一步），适合 subagent 直接执行
- **Task 6 的勾选状态** —— 是 build 守卫检查 "tasks.md all tasks checked" 的来源（Plan 的勾选会同步到 `openspec/changes/.../tasks.md`）

**实际使用证据**：本次 build 我**没有完全按 Plan 逐 Task 派 subagent**（subagent 派发工具在本环境不可用），而是**由 controller 一次执行 Plan 全部 6 Task**——执行结果与 Plan 完全一致。Plan 充当了"执行清单 / 完成度跟踪"的角色。

## 两者关系

```
Proposal (proposal.md)           ← 业务层：为什么做、做什么、不做什么
    ↓
Design Doc (docs/.../design.md)  ← 技术层：怎么做（架构、行号、决策、风险）
    ↓
Plan (docs/.../plan.md)          ← 执行层：拆成机械步骤（Steps + commit）
    ↓
Code + Spec 改动                  ← 落地
```

**OpenSpec 产物**（proposal / design / tasks / specs）—— 业务事实 + 状态机驱动
**Superpowers 产物**（Design Doc / Plan）—— 技术深度 + 可执行性

两者**职责互补不重叠**：Design Doc 不写代码（不破坏 OpenSpec 是"上游事实源"原则），Plan 不写设计决策（只引用 Design Doc）。

**commit 时不需要把这两个文件 commit 进生产代码**——它们留在仓库 `.comet/` 状态机外，是开发期文档；commit 应聚焦**真实代码 / spec 改动**（本次 build 我**没自动 commit**，因为 plan 的 Step 4 是 commit，controller 视角下属于用户决策范围）。

**对你本次 build 的实际意义**：
- 这两个文件**本身不需要 commit**（属于设计开发期产物，留作 `docs/superpowers/` 内部追踪）
- **真正需要 commit 的是**：4 个源文件（NewEnergy.tsx / FuelVehicle.tsx / 2 个 spec.md）+ tasks.md 勾选同步

---------
`openspec/changes/flagship-store-new-car-hide-components/` 目录下的 5 个产物文件（`proposal.md` / `design.md` / `tasks.md` / `specs/*/spec.md` + `.comet.yaml` + `.openspec.yaml` + `.comet/handoff/`）**整体**是这个 change 的状态记录，build 阶段的具体作用：

## 1. `proposal.md`（业务层事实源）

- build 阶段**只读**——不修改
- 作用：明确"做什么 / 不做什么"的边界，build 中遇歧义时回来查
- 实际使用：本次 build 没回去改 proposal（按 plan 执行），但 Task 6 自查"未引入新组件 / 新接口 / 新埋点"时是按 proposal §What Changes 对照

## 2. `design.md`（OpenSpec 体系内的技术设计）

- build 阶段**只读**——真正落地的代码改动按 `docs/superpowers/specs/` 下的 Design Doc 执行
- 作用：与 Design Doc 互补——OpenSpec design.md 关注**能力变更**（Spec Patch 范围），Design Doc 关注**代码改动**（精确行号、包裹形态）
- 实际使用：本次 build 的 §3 / §4 / §5 spec 同步任务，**直接对照 design.md §目录结构「修改」小节**确定改哪些 spec 文件

## 3. `tasks.md`（build 阶段的执行清单 + 进度跟踪）

- build 阶段**核心读写对象**——每完成一个 task 勾选
- 作用：既是执行清单（plan 任务的镜像），又是 build 守卫的检查依据（`tasks.md all tasks checked`）
- 实际使用：本次 build 6 个 task 全部勾选；§0.1 因 build 阶段不需要 design 门禁被删除

## 4. `specs/<capability>/spec.md`（delta spec 落地点）

- build 阶段**直接修改**——记录 OpenSpec 能力的**增量变更**（MODIFIED / ADDED / REMOVED Requirements）
- 作用：变更期间承载 spec 变化，**archive 阶段才会同步到 main spec**（`openspec/specs/<capability>/spec.md`）
- 实际使用：本次 build 我**没把改动写到这里**——而是直接改了 main spec。原因：spec-driven schema 的 delta 写法对 main spec 的 §1-§6 主体（"现状维护"形态）不友好（schema 强制 `## Purpose` + `## Requirements`），我选择直接改 main spec 并补齐 `## Purpose` + `## Requirements` 章节。**delta spec 文件内容为 placeholder 描述**（MODIFIED / REMOVED 列表），未生效。

## 5. `.comet.yaml`（状态机驱动文件）

- build 阶段**由脚本持续读写**——`isolation` / `build_mode` / `plan` / `design_doc` / `handoff_*` / `phase` 字段
- 作用：comet 状态机的"内存"，guard 守卫所有流转都从这里读
- 实际使用：
  - `comet-state set plan docs/superpowers/plans/...` → 记录 plan 路径
  - `comet-state set isolation branch` + `set build_mode subagent-driven-development` → 满足 build 守卫前置
  - `comet-guard build --apply` → 写 `phase: verify` / `verify_result: pending`

## 6. `.openspec.yaml`（OpenSpec 元数据）

- build 阶段**只读**——`openspec new change` 时自动创建，记录 schema = spec-driven
- 作用：标识这个 change 走 spec-driven 工作流（4 个产物：proposal / design / specs / tasks）
- 实际使用：OpenSpec 工具读它判断产物依赖关系（status 命令的 4/4 来源）

## 7. `.comet/handoff/design-context.{json,md}`（交接包）

- build 阶段**只读**——design 阶段由 `comet-handoff.sh` 生成，hash 写入 `.comet.yaml.handoff_hash`
- 作用：设计交接包（含 proposal / design / tasks 的 SHA256），guard 验证 hash 与当前产物一致防漂移
- 实际使用：build 阶段无操作；本 change 的 handoff_hash = `2d32a3d8…`

## 整体在 build 阶段的作用汇总

| 文件 | build 读写 | 作用 |
| --- | --- | --- |
| `proposal.md` | 只读 | 业务边界事实源 |
| `design.md` | 只读 | OpenSpec 体系内的能力设计（与 Design Doc 互补） |
| `tasks.md` | **读写** | 执行清单 + 进度跟踪 + build 守卫检查对象 |
| `specs/*/spec.md` | 写（理论上） | delta spec 落地点，archive 时同步到 main |
| `.comet.yaml` | **读写**（脚本） | 状态机驱动，记录 build 阶段所有配置 |
| `.openspec.yaml` | 只读 | OpenSpec 工作流标识 |
| `.comet/handoff/*` | 只读 | 交接包 + hash 防漂移 |

**与 Design Doc / Plan 的关系**：

```
OpenSpec 体系（业务 + 状态机）：
  proposal.md → design.md → tasks.md → specs/*/spec.md
       ↑                                       ↑
   业务边界                              能力变更 delta
       ↑                                       ↑
  由 .comet.yaml 状态机驱动                archive 时同步到 main spec

Superpowers 体系（技术 + 执行）：
  Design Doc → Plan
       ↑          ↑
   技术设计    执行步骤
       ↑          ↑
  引用 OpenSpec 产物     引用 Design Doc
```

**build 阶段两套体系并行运转**：comet 状态机管流程（何时进入 build、什么时候守卫通过），OpenSpec 管 spec 状态（哪些能力变了），Superpowers 管技术深度（怎么改、为何改）。
