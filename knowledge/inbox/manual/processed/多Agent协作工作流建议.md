# 多Agent协作工作流建议

## 核心建议

### 1. 缺少"上下文共享机制"
- Orchestrator 应生成"任务上下文卡"（任务ID、约束、历史决策）
- 各守卫的结论应以结构化格式汇总
- 实现Agent在编码前必须显式确认理解了所有守卫结论

### 2. 验收标准不够量化
- 定义"BR命中率"指标（如 85%以上为可接受）
- blocking问题 vs 建议性问题要分类

### 3. 并行守卫的同步点不明确
- 补充守卫间的依赖关系
- 超时/冲突时的仲裁规则

### 4. 缺少"失败案例库"和回溯机制
- 记录 Phase 升级的真实案例
- 定义"降级回滚"的流程

### 5. 前置检查清单
- □ 任务是否有明确的验收标准？
- □ 涉及的文件是否有最新的类型定义？
- □ 是否需要跨端测试？
- □ 是否涉及Native接口调用？
- □ 新增文件是否有路径规范文档？

### 6. 模式选择决策树
```
是否涉及2个以上域？ → Yes → Phase 2
是否涉及多端差异？ → Yes → Phase 2
是否是关键链路？   → Yes → Phase 2
→ Phase 1
```

## 守卫的本质
守卫不是检查"已经改完的代码"，而是检查**"这个需求意图"是否安全**。在动手改代码前，先做预防性风险扫描和合规审查。

## AI参与度提升
- 自动化模式推荐（推荐Phase等级）
- 自动生成上下文卡
- 智能守卫验证（命名规范、BR对标、风险列表）
- 汇总报告自动生成（PR描述+验收清单）
