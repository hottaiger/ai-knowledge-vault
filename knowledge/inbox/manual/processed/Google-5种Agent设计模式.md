# 5种Agent技能设计模式

> 来源：Google Cloud Tech Twitter/X
> 作者：Shubham Saboo & Lavini Gam
> 日期：2026-03-22

---

## ✨ 核心要点

1. **工具封装模式**：让Agent成为类库专家
2. **生成器模式**：基于模板生成结构化文档
3. **审查器模式**：按严重程度为代码评分
4. **反向交互模式**：先沟通需求，再执行
5. **流水线模式**：通过检查点实现严格工作流

---

## 模式一：工具封装（Tool Wrapper）

**作用**：为Agent提供特定类库的按需上下文信息

**原理**：
- 无需将API规范硬编码到系统提示词
- 封装为Skill，仅在处理该技术相关任务时加载上下文
- 监听用户提示词中的关键词，动态加载references/

**示例**：
```
# skills/api-expert/SKILL.md
---
名称：api-expert
描述：FastAPI开发的最佳实践与规范
---
你是FastAPI开发领域的专家。

## 核心规范
加载references/conventions.md文件。

## 代码审查流程
1. 加载规范参考文档
2. 逐一核对用户代码是否符合规范
3. 标注违规并给出修复建议

## 代码编写流程
1. 加载规范参考文档
2. 严格遵循所有规范
3. 为所有函数签名添加类型注解
```

## 模式二：生成器（Generator）

**作用**：保证输出的一致性

**原理**：
- 通过标准化的"填空式"流程
- `assets/` 存放输出模板
- `references/` 存放格式指南
- 技能指令协调模板加载、指南读取、变量询问

**示例**：
```
# skills/report-generator/SKILL.md
---
名称：report-generator
描述：生成Markdown格式的结构化技术报告
---

你是技术报告生成助手，请按以下步骤执行：

步骤1：加载references/style-guide.md
步骤2：加载assets/report-template.md
步骤3：向用户询问缺失信息：
  - 报告主题
  - 核心结论或数据要点
  - 目标受众
步骤4：遵循格式指南填充模板
步骤5：返回完成的Markdown文档
```

## 模式三：审查器（Reviewer）

**作用**：将"审查内容"与"审查方式"分离

**原理**：
- 模块化评审标准存放在references/
- 用户提交代码后，Agent加载清单
- 按严重程度分类输出

**严重程度等级**：
- 错误（必须修复）
- 警告（建议修复）
- 提示（可参考优化）

**示例**：
```
# skills/code-reviewer/SKILL.md
---
名称：code-reviewer
描述：审查Python代码的质量、格式及常见漏洞
---

你是Python代码审查专家，请按以下流程：

步骤1：加载references/review-checklist.md
步骤2：明确代码用途后再评审
步骤3：发现违规项时：
  - 标注行号
  - 划分严重程度
  - 说明原因
  - 给出修复方案

步骤4：生成结构化审查报告：
- 审查摘要
- 问题发现（按严重程度）
- 评分（1-10分）
- 三大核心建议
```

## 模式四：反向交互（Inversion）

**作用**：颠覆"用户发指令、Agent执行"的固有逻辑

**原理**：
- Agent以"面试官"身份主动沟通
- 强制Agent先收集完整上下文
- 严格的阶段划分和门控提示
- **所有阶段完成前，禁止开始开发**

**示例**：
```
# skills/project-planner/SKILL.md
---
名称：project-planner
描述：通过结构化问题收集需求，再制定规划
---

你正在开展结构化的需求调研。
**所有阶段完成前，禁止开始任何开发或设计工作。**

## 第一阶段——需求挖掘
一次提出一个问题，等待用户回答后再继续：
1. 项目为用户解决了什么核心问题？
2. 核心用户是谁？技术水平如何？
3. 预期规模？（用户量、数据量、请求率）

## 第二阶段——技术约束
4. 使用哪种部署环境？
5. 技术栈要求或偏好？
6. 硬性要求？（延迟、可用率、合规性、预算）

## 第三阶段——规划生成
1. 加载assets/plan-template.md
2. 填充模板的每个章节
3. 展示完整规划
4. 询问用户反馈并迭代优化
```

## 模式五：流水线（Pipeline）

**作用**：通过硬性检查点，实现严格的顺序化工作流

**原理**：
- 技能指令本身就是工作流定义
- 设置明确的门控条件
- 必须获得用户确认才能进入下一阶段
- 任何步骤失败，禁止继续推进

**示例**：
```
# skills/doc-pipeline/SKILL.md
---
名称：doc-pipeline
描述：从Python源码生成接口文档的多步骤流水线
---

你正在执行文档生成流水线。
**不得跳过步骤，若某一步执行失败，禁止继续推进。**

## 步骤1——代码解析
分析Python代码，提取所有公共类和函数。
询问用户确认是否完整。

## 步骤2——文档字符串生成
- 加载references/docstring-style.md
- 生成文档字符串
- **用户确认前，禁止进入步骤3**

## 步骤3——文档整合
加载assets/api-doc-template.md
整合为单个接口参考文档。

## 步骤4——质量检查
- 所有公共符号已完成文档化
- 所有参数已标注类型
- 每个函数至少一个使用示例
```

## 模式选择决策树

| 场景 | 推荐模式 |
|------|----------|
| 需要调用特定类库/框架 | 工具封装 |
| 生成格式不一致 | 生成器 |
| 代码审查/质量检查 | 审查器 |
| 需求不明确，容易返工 | 反向交互 |
| 复杂任务，容易遗漏步骤 | 流水线 |

---

## 模式的组合使用

**可以灵活组合**：

- 流水线 + 审查器：最后一步对生成结果二次校验
- 生成器 + 反向交互：模板填充前先收集所有变量

借助ADK的SkillToolset和渐进式信息披露能力，Agent仅在实际需要时消耗上下文令牌。

## 💡 我的思考

这5种模式解决了Skills"怎么写"的问题：

- **工具封装**：解决"知识获取"
- **生成器**：解决"输出一致"
- **审查器**：解决"质量把控"
- **反向交互**：解决"需求理解"
- **流水线**：解决"流程规范"

之前学的Skills框架是"形式"，这5种模式是"内容逻辑"。
