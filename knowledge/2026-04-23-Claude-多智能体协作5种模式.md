# Claude 多智能体协作5种模式

> 本文整理自慢学AI对 Claude Blog「Multi agent coordination patterns」的解读视频。原文：https://claude.com/blog/multi-agent-coordination-patterns

## 核心建议

从最简单的模式开始，看它在哪里撑不住，再往上演化。复杂模式的代价是真实的——调试难、追踪难、出了问题不知道从哪查起。

## 五种模式一览

| 模式 | 协作方式 | 比喻 |
|------|---------|------|
| **Generator-Verifier** 生成验证器 | 生成↔验收，迭代直到通过 | 一个人起草，另一个人拿着验收单逐条过 |
| **Orchestrator-Subagent** 编排者子agent | 主agent拆任务→派→收结果 | 总包经理临时叫几位专家来会诊，看完结果收回来 |
| **Agent Teams** agent团队 | 协调者带多个长期worker | 工地上的几支常驻班组，各守一块区域，时间越久越熟 |
| **Message Bus** 消息总线 | 通过发布/订阅事件协作，router分发 | 医院分诊台：来了什么情况，一站交给谁 |
| **Shared State** 共享状态 | 所有agent直接读写同一个共享存储 | 专案组围着同一块白板办案，谁补一条线索，别人立刻能接着查 |

## 模式详解

---

### 1. Generator-Verifier（生成验证器）

**核心思路：** 把"做出来"和"判过关"拆成两步。一个agent生成结果，另一个按标准验收；验不过就打回去改，改完再验，直到通过或达到最大轮数。

**适用场景：** 代码生成+测试、合规审查、按 rubric 打分——通过标准可以被写清楚的任务。

**三个难点：**

1. **验证器没有明确标准** — 只说"你帮我看看好不好"只是走形式
2. **生成和评估本身一样难** — 判断文案有没有说服力，和写出好文案一样难，第二个 agent 没有站在更客观的位置
3. **循环不收敛** — 一直在生成打回再生成，但每一轮都没真正逼近通过。必须配最大迭代次数 + fallback

---

### 2. Orchestrator-Subagent（编排者子agent）

**核心思路：** 主agent理解总任务、拆分子任务、派出去、收回来、拼成完整答案。子agent各自开独立上下文窗口，做完只带提炼结果回来。

**适用场景：** 能被清楚拆解、边界明确、彼此依赖不重的任务（如代码审查同时检查安全/测试/风格/架构）。

**两个难点：**

1. **主agent成信息瓶颈** — A子agent的发现影响B的工作，需要先回到主agent再转述，细节容易损耗
2. **速度不一定更快** — 如果没有显式并行化，子agent还是一个接一个跑，token成本上去但吞吐量没提

---

### 3. Agent Teams（agent团队）

**核心思路：** 和上一种的区别只有一个——worker长期存在，反复处理自己负责的那类工作，做完不结束。随着时间推移，worker对局部系统越来越熟。

**真正买到的：** 除了并行，还有长期上下文积累。一次性子agent很难复制这种局部经验。

**三个难点：**

1. **独立性是硬前提** — worker改的东西会持续影响另一个但不能及时共享中间发现，结果就打架
2. **完成检测更难** — 不同worker节奏完全不同，有人2分钟有人20分钟，协调者必须处理不同步
3. **共享资源冲突** — 两个worker改同一个文件或做出互不兼容的变更。最考验能不能把任务切成真正相对自治的分区

---

### 4. Message Bus（消息总线）

**核心思路：** agent之间通过发布和订阅事件协作，由router负责分发。关键不只是大家通过消息协作，还包括专门的路由层决定什么事件该送到谁手里。

**适用场景：** 事件驱动的pipeline流程，没有被提前完全写死，会随着事件一路往前长出来。扩展性强，新agent只需订阅topic就能接入。

**两个难点：**

1. **追踪变难** — 告警可能在5个agent之间触发一串连锁事件，需要非常仔细的日志和事件关联
2. **路由准确性非常关键** — 如果router把事件分错或丢了一个，系统不报错但该处理的没人处理。如果router还用LLM做语义路由，失败模式会更多

---

### 5. Shared State（共享状态）

**核心思路：** 没有中央协调者，也没有router。所有agent直接读写同一个共享存储（数据库/文件系统/共享文档/工作记忆）。谁发现了什么就直接写进去，别人基于发现继续推进。

**最大优势：** 共享存储慢慢长成一个不断演化的知识空间。某个agent挂了不代表整个系统立刻停摆。

**两个难点（两类风险）：**

**工程问题：** 重复劳动（两个agent调查了同一条线索）、并发写冲突。靠加锁、版本控制、任务分区缓解。

**行为问题：** Reactive loop — A写入发现，B补回应，A又看见B的回应再补一条……系统始终很忙，token始终在烧，但答案没有真正收敛。

**最重要的设计点：终止条件必须提前设计。** 什么时候停？时间到停？连续几轮没有新发现停？还是专门指定一个agent判断够了可以停了？

## 三对容易混淆的模式

### 编排者子agent vs Agent Teams

都像一个人带多个worker，但感觉不同：
- 编排者子agent = 总包经理临时叫专家来会诊，看完这一轮就散
- Agent Teams = 工地长期驻场的班组，一直守着各自那块区域

**判断标准：** worker做完这一轮是结束还是继续活着

### Agent Teams vs Shared State

都像多个agent长期协作，但协作方式不同：
- Agent Teams = 几个小组各查各的，最后把结果交回来汇总
- Shared State = 所有人围着同一块按键白板，谁补了线索别人立刻顺着查

**判断标准：** 中间发现要不要实时互相影响

### Message Bus vs Shared State

都弱化了传统总指挥，但中心不同：
- Message Bus = 医院分诊台，来了什么情况下一站该交给谁
- Shared State = 多个医生围着同一份不断更新的病例协作

**判断标准：** 系统围绕的是事件流还是共享知识

## 选型提示

- 从最简单的模式开始，看它在哪里撑不住再往上演化
- 复杂模式的代价是真实的：调试难、追踪难
- 选错了死法各异：token烧但不收敛、互相覆盖、线上一挂全瘫、消息路由错什么都没处理、reactive loop看似在跑实际没收敛
