# Figma MCP 缓存设计稿——get_design_context 返回结构解析

> [!quote] 一句话理解
> Figma MCP 的 `get_design_context` 返回的不是图片，是 **React + Tailwind 的参考实现代码 + 提示原文**——图片资源是临时 URL（约 7 天有效），需要立刻缓存到本地。

## 核心观点

- **返回结构 = 代码 + 提示**：`get_design_context` 返回 React 组件代码（含 `data-node-id` 注入），同时附带 4 条提示原文。代码不能直接用，提示信息很关键。
- **图片资源临时性**：返回里所有 `https://www.figma.com/api/mcp/asset/<uuid>` 是**约 7 天有效**的临时 URL，必须立刻缓存（下载到项目 assets 目录或者上传到自己的 CDN）。过 7 天再请求会 404。
- **不能默认 Tailwind**：返回代码用 Tailwind 写的，但提示明确说"不要默认安装 Tailwind"——需要先评估目标项目技术栈，把 className 翻译到目标样式系统（CSS Modules / styled-components / Taro 的 less 等）。
- **`data-node-id` 是双向定位锚**：返回代码里每个 DOM 节点都注入了 `data-node-id`（对应 Figma 节点 ID），方便反向追到设计稿具体位置做调整。
- **典型工作流**：调 MCP → 拿到代码 + 图片 URL → 立刻批量下载图片资源（避免过期）→ 评估技术栈差异 → 改造样式系统 → 整合到目标项目。

## 我的思考

MCP 这种"返回参考实现而非成品"的模式是工具型协议的设计哲学——**只暴露数据 + 协议，不替你做风格决策**。这跟 [[concepts/MCP工具链]] 的核心思想一致：MCP 提供原料，转换是 Agent / 人的事。
